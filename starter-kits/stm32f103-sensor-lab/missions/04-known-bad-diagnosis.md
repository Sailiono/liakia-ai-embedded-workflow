# Mission 04: Known-Bad Diagnosis

This is the core starter mission. The user flashes a broken application layer, observes an automated gate failure, and uses evidence to locate the problem.

## Principle

The known-bad bug stays in the application layer. It does not modify the user-generated IOC or HAL driver code.

This demonstrates:

```text
the low-level project is generated by the user
application-layer issues can be caught by gates
AI analysis is grounded in logs, protocol frames, and registers
the fix can remain minimal
```

## Recommended First Case: Case B

Use:

```text
app-layer/known-bad/case_b_bmp280_calibration/
```

Expected symptoms:

```text
sensor id PASS
raw calibration bytes readable
temperature data-quality gate FAIL
```

This is better than a wrong I2C address for demonstration: the sensor appears readable, but the data is still not trustworthy.

## Evidence To Collect

At minimum:

```text
version output
sensor id output
sensor read output
telemetry once output
raw calibration bytes
raw adc temperature
compensated temperature
data quality gate result
```

If register probing is available, record:

```text
RCC_APB1ENR
GPIOB_CRL
GPIOB_IDR
I2C1_CR1
I2C1_SR1
I2C1_SR2
USART1_BRR
```

## AI Input Requirements

Do not send only "the temperature is wrong" to the AI. Provide:

```text
Hardware: STM32F103C8T6 + BMP280
Interface: I2C1 PB6/PB7 100 kHz
Symptom: chip id PASS, compensated data out of range
Logs: paste sensor id / sensor read / telemetry once
Code: paste calibration read and compensation functions
Constraint: inspect application layer first; do not blame hardware unless evidence supports it
```

## Expected Diagnosis Direction

Case B is not mainly a bus problem. Focus on:

- BMP280 calibration little-endian assembly;
- signed / unsigned types;
- intermediate integer width;
- whether the compensation formula matches the datasheet;
- raw ADC looks valid but compensated value is invalid.
