# Mission 04: Known-Bad Diagnosis

This is the core starter mission. The user imports a broken application-layer case, flashes it, observes a gate failure, and uses evidence to locate the problem.

## Principle

The known-bad code is supplied as a case folder. The user still owns the CubeMX IOC, HAL initialization, wiring, build, and flash steps.

This demonstrates:

```text
the low-level project is generated by the user
the broken code is imported deliberately
automated gates catch the failure
AI analysis is grounded in logs, protocol frames, and registers
the fix remains minimal and reviewable
```

## Recommended First Case

Use the Case B folder:

```text
known-bad-cases/case-b-bmp280-calibration/
```

Follow its practice guide first:

[Case B practice guide](../known-bad-cases/case-b-bmp280-calibration/README.md)

Do not open the answer key until the failing evidence package exists and you have tried the AI diagnosis prompt.

## Evidence To Collect

At minimum:

```text
version output
diag i2c output
sensor id output
sensor read output
telemetry once output
raw calibration or raw protocol bytes
gate result
```

If register probing is available, record:

```text
RCC_APB1ENR
GPIOB_CRL
GPIOB_IDR
I2C1_CR1
I2C1_SR1
I2C1_SR2
USART1_BRR
```

## AI Input Requirements

Do not send only a single symptom sentence to the AI. Provide:

```text
Hardware: STM32F103C8T6 + BMP280
Interface: I2C1 PB6/PB7 100 kHz
Imported case folder: known-bad-cases/case-b-bmp280-calibration
Logs: paste diag i2c / sensor id / sensor read / telemetry once
Code: paste only the imported app-layer file or the suspicious functions
Constraint: reason from evidence first; do not blame hardware unless logs support it
```

Read the case answer key only after this diagnosis attempt.
