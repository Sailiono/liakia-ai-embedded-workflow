before / after

Liakia 改变的是验证和交接方式，不是只多一个脚本。

它把原来散落在 IDE、烧录工具、串口窗口、聊天记录和人工记忆里的步骤，收敛成可以复跑、可审查、失败也留痕的交付链路。
before liakia

手工台架流程

  • 工程师手动 build 和烧录。
  • 串口测试靠记忆和临场操作。
  • 失败日志散落在聊天记录和截图里。
  • 修复后没有稳定 regression gate。
  • 远程测试台依赖现场人员反复配合。
after liakia

证据驱动闭环

  • 一条 baseline runner 串起 build / flash / test / probe。
  • 失败时也生成 manifest、logs 和 summary。
  • USB CDC reset recovery 变成必跑 gate。
  • RTCM CRC BAD = 0 可以直接审查。
  • remote HIL 从测试台回传 evidence package。

adopt it

接入客户项目时，不替换现有栈，只封装交付流程。

Liakia 的目标是把已有工程的 build、flash、test、probe 和 evidence 输出描述清楚，再用 adapter-driven runner 调度，而不是强推新的 IDE、RTOS 或驱动框架。
day 1

Build / Artifact

确认构建命令、输出 ELF/HEX/BIN 和 artifact hash。

project-adapter.json
day 2

Flash / Verify

接入 STM32CubeProgrammer transcript，避免口头描述“已经烧进去了”。

SWD / verify / reset
day 3-4

Serial / Protocol Gates

把 Shell、业务命令、协议输出、CRC/帧解析沉淀成自动 gate。

tests / parser / fail gate
day 5

Evidence / Handoff

生成 manifest、summary、handoff report，并明确哪些结论需要人工审核。
