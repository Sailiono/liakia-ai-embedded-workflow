前后对比

Liakia 改变的是验证和交接方式，不是只多一个脚本。

它把原来散落在 IDE、烧录工具、串口窗口、聊天记录和人工记忆里的步骤，收敛成可以复跑、可审查、失败也留痕的交付链路。
接入前

手工台架流程

  • 工程师手动 build 和烧录。
  • 串口测试靠记忆和临场操作。
  • 失败日志散落在聊天记录和截图里。
  • 修复后没有稳定的回归验证。
  • 远程测试台依赖现场人员反复配合。
接入后

证据驱动闭环

  • 一条基线脚本串起编译、烧录、测试和寄存器快照。
  • 失败时也生成运行清单、日志和摘要。
  • USB CDC reset 恢复变成必跑检查。
  • RTCM CRC BAD = 0 可以直接审查。
  • 远程测试台回传完整证据包。

核心分工

AI 是油门，工程师是方向盘和刹车。

这不是“AI 自动写固件”的展示，而是工程师把 AI 接进真实硬件闭环。工程师提供方向、判断和风险边界，AI 负责按流程快速执行、补齐遗漏、整理证据，并把跑通的流程沉淀成可复用 skill。
AI 加速执行

油门

  • 按流程快速执行 build / flash / test。
  • 读取日志、摘要和运行清单，补充容易遗漏的排查项。
  • 生成诊断材料、报告草稿和下一步建议。
  • 把已验证流程保存为 skill，后续调用同款操作。
工程师掌舵管控

方向盘和刹车

  • 定义目标、风险边界和什么结果才算 PASS。
  • 决定哪些故障必须变成长期 regression gate。
  • 控制烧录、reset、寄存器读取等硬件操作边界。
  • 审查证据是否足够，负责最终验收和交付判断。
工程师定义方法，AI 执行方法；工程师沉淀流程，AI 复用流程。 速度来自 AI，可信度来自工程边界和证据闭环。

能力栈

Liakia 展示的不是模型能力，而是工程师把复杂交付系统化的能力。

01

问题建模

把“reset 后偶发不可用”定义成可重复验证的 reset recovery gate。

02

台架自动化

把 CMake、CubeProgrammer、COM 口、RTCM 和 reset 恢复串成基线 runner。

03

协议验证

用 RTCM parser 和 CRC BAD = 0 证明输出真的可用，不靠肉眼看日志。

04

寄存器诊断

选择 RCC、GPIO、USART、USB 等只读寄存器作为低层证据。

05

证据工程

让 PASS 和 FAIL 都有 manifest、logs、summary 和 handoff 入口。

06

AI 边界

定义 pre-flash、failure triage 和人审停止条件，避免自动化越权。

07

可迁移抽象

用 project-adapter 和 workflow-template 把 dpiny-RTK 抽象成可接入流程。

08

流程沉淀

把一次成功的复杂操作整理成 skill，让 AI 后续稳定复用。
