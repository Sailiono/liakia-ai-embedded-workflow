starter lab

Starter Lab 的核心是“先自己排查，再让 AI 排查，最后看答案”。

用户先搭好 STM32F103C8T6 + BMP280 baseline，再从四个 known-bad case 中选择一个导入。每个 case 都有独立 broken app-layer、操作指南和答案文件；答案默认放在最后，避免一开始泄题。
01

先跑通正常 baseline

用户用 CubeMX 创建 STM32F103C8T6 工程，配置 SWD、USART1、I2C1 和 GPIO。Liakia 只给配置检查点，不替用户生成完整工程。

IOC / HAL / SWD / USART1 / I2C1
02

选择四级故障练习

Case A-D 按难度递进。推荐从 Case B 的 BMP280 数据异常开始，再进入 I2C reset、Flash 持久化和 UART DMA/IDLE 边界问题。

Case B -> Case A -> Case D -> Case C
03

先人工定位故障

导入 broken app-layer 后，用户自己观察 gate FAIL、串口日志和 manifest，记录失败现象、假设、排除项和耗时。

symptom / hypothesis / elapsed time
04

再让 AI 用同一证据排查

runner 生成 logs、manifest 和 `ai_prompt.md`。用户让 AI 基于证据定位根因，再和自己的路径对比，最后打开 ANSWER 验证。

manual vs AI / answer last / regression PASS
人工优先先记录自己看到的现象、推理路径和耗时。
AI 第二把同一 evidence 交给 AI，要求只基于日志和 gate 结果分析。
答案最后完成对比后再打开 ANSWER，用来校验而不是提前泄题。

roi model

ROI 不是只讲 API 便宜，而是讲周期、风险和交接歧义被压缩。

traditional estimate

ai-assisted compression

3 人天 + 约 ¥10 API 消耗

反馈编译 warning/error 快速分析
测试runner、parser 和 gate 复用

guardrails

AI 不替代工程判断

  • 原理图、供电、隔离、安全边界仍需人工审核。
  • EMC/ESD、协议边界、量产风险不能靠模型拍板。
  • 最终代码 review 和验收签字保留在人审闭环内。
