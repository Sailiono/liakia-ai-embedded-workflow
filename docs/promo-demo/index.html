failure-to-fix cases

客户真正会追问的是：出问题时，这套流程怎么缩短定位时间？

case 01 USART Clock Missing

现象：编译和烧录通过，但串口通道无输出。

symptom register probe AI summary engineer confirm regression PASS
  • 读取 RCC_APB1ENR、GPIO AF、USART CR1/BRR。
  • 优先排除物理接线和波特率问题。
  • 修复后跑 Shell 与 RTCM 回归。
查看完整复盘
case 02 RS422 DE Timing

现象：RTCM 输出存在间歇性丢帧或尾字节异常。

capture DMA state DE edge minimal fix CRC BAD: 0
  • 把 DMA 完成、DE 控制和串口输出日志放在同一时间线。
  • 定位方向从协议层缩小到收发器方向控制。
  • 用 RTCM parser 做修复后证明。
查看完整复盘
case 03 RTCM CRC Validation

现象：有输出但业务侧怀疑数据链路不可信。

frame scan type count crc check artifact handoff
  • 解析 RTCM3 帧头、长度、payload 和 CRC24Q。
  • 统计消息类型覆盖与异常帧数量。
  • 把“能输出”升级成“可验真”。
查看完整复盘
case 04 USB CDC Reset Recovery

现象：Windows 能看到 USB CDC 口，但 reset 后 Shell 偶发不可用。

probe fail reset gate shell version recover baseline PASS
  • 把“串口存在但 Shell 不响应”变成自动化 gate。
  • reset 前后都发送 version 验证 CDC Shell 可用。
  • 与 build、flash、RTCM CRC 同一 baseline 归档。
